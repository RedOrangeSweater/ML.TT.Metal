## /sdlc.audit — аудит docs и SDLC-артефактов

Русский, структура без воды. Цель: согласованность навигации, SoT, статусов, спек, QA, run configs, Mermaid.

### Input

Опционально: путь к `TASK_*`, описание изменений (1–5 строк) или список путей. **Пусто** → полный audit: сначала фаза 0 (упрощение + код SoT, без изменения planned-доков), затем слои 1–6.

### Режимы

- **Без параметров**: сначала **фаза 0** (в начале — сформулировать, какими диалектами, трансформациями, запросами пользоваться; **только из общего репо** _KG_MLIR; при отсутствии — создать там; при необходимости изменить — предложить правку; затем упрощение формулировок + код SoT; сохранять запланированные доки; учёт KG/MLIR и заметок после абзацев/диаграмм), затем слои 1–6 по порядку; каждый слой улучшай перед переходом к следующему.
- **С параметрами**: быстрый audit только по затронутым частям.

### Чеклист (по умолчанию)

- SoT + start here: `docs/README.md` → `docs/sdlc/00_Main/00_Status/00_projects_graph.md`.
- Статус: `00_projects_graph.md`, `01_Kanban.md` — валидны, без битых путей.
- Run configs: доки «как запускать» ↔ `.vscode/{launch.json,tasks.json}`.

### Output

- **Фаза 0** (если выполнялась): **диалекты/трансформации/запросы** — перечень из общего репо (_KG_MLIR), что использовать; что создать или изменить в общем репо. Упрощение/код SoT — предложения по файлам; оставлено без изменений (planned); KG/MLIR — замечания по стилю/заметкам.
- **SoT** + start here.
- **Changed surface**: 3–10 буллетов (что менялось).
- **Required SDLC updates**: Task context, BA, Arch, Testing, Status — или «не требуется».
- **Gaps / risks**: 1–5 пунктов.
- **Next actions**: 3–7 пунктов «сделай X в файле Y».

### Constraints

Фокус на согласованности; без широких рефакторов. Коммит/пуш — atomic workflow, без auto-push.

**Код — SoT**: при расхождении доков с кодом предлагать обновлять документацию из кода (исходники, конфиги, тесты). Запланированные архитектурные/BA/спеки не перезаписывать из кода (документ целевой, planned).

**Стиль и KG/MLIR**: простой язык, коротко, по делу; контент будет транслирован в KG (понятия, связи, эффекты), текст в конце — LLM из MLIR (пайплайн §3.2). При аудите предлагать упрощение/сокращение формулировок. **Диалекты, трансформации, запросы — только из общего репо** (docs/sdlc/_KG_MLIR или .cursor/artifacts_mlir_graphdb); при отсутствии — предложить создать в _KG_MLIR; при необходимости — предложить изменить существующий. **Путь к артефактам**: канон — _KG_MLIR или .cursor/artifacts_mlir_graphdb; ссылки на прежний путь Artifacts — некорректны (каталог удалён), предлагать заменить на _KG_MLIR. Цель: итеративное улучшение при каждом вызове. При наличии пайплайна Doc–MLIR–GraphDB проверять заметки после абзацев/диаграмм (диалект, MLIR).
